# FG-IG-Scan 实现路线与版本演进记录

## 0. 文档目的

这份文档用于记录 FG-IG-Scan 创新点从提出、实现、暴露问题到迭代修正的全过程，作为后续论文写作、组会汇报、实验回溯和版本管理依据。

从本版本开始，FG-IG-Scan 每次发生方法级演进，都单独记为一个版本：v1、v2、v3、v4...

要求：

1. 每个版本必须写清楚核心改动。
2. 每个版本必须写清楚为什么要改。
3. 每个版本必须记录暴露出的关键问题。
4. 每个版本必须说明是否进入正式训练验证。

---

## 1. 创新点目标

FG-IG-Scan 的目标不是简单“改一下扫描顺序”，而是解决遥感实例分割中以下核心矛盾：

1. 前景区域稀疏，固定扫描顺序会让 SSM 隐状态首先被大量背景占据。
2. 小目标在长序列建模中容易被大面积背景稀释。
3. 遥感图像中的目标分布具有明显空间非均匀性，固定扫描路径无法体现“先处理关键区域”的需求。

因此，FG-IG-Scan 的主线目标始终是：

让扫描顺序从固定路径，演进为受前景重要性引导的动态优先级扫描。

---

## 2. v1：区域 2D 重排版本

### 2.1 核心思路

v1 的出发点是：

1. 先预测区域重要性。
2. 在 2D 空间上对区域进行重排。
3. 再把重排后的特征送入官方 VMamba 的 selective scan 路径。

简化理解：

先改空间布局，再交给原始扫描器去扫。

### 2.2 实现特征

v1 主要采用的是：

1. 区域级重要性预测。
2. 2D 区域 permute / inverse permute。
3. 外部包裹官方 forward_core，不直接改内部 cross_scan。

### 2.3 暴露的问题

这是 FG-IG-Scan 第一次真正暴露核心缺陷的版本。

关键问题在于：

1. 它不是真正的 1D 扫描顺序控制。
2. 它只是 2D 区域重排，但后面仍然调用官方固定 cross_scan。
3. 官方 4-path 扫描与 merge 会冲淡区域优先级的效果。
4. 因此，重排后的“优先顺序”并没有被充分保留下来。

### 2.4 结论

v1 证明了一件事：

只在 2D 布局层面重排，而不真正控制 scan sequence，本质上是不够的。

因此 v1 被判定为：

概念上接近目标，但实现路径偏弱，无法充分体现“区域优先扫描”的真实作用。

---

## 3. v2：区域优先级 1D 重排版本

### 3.1 为什么从 v1 升级到 v2

v1 的主要问题不是“没有 importance”，而是 importance 对真正扫描序列的控制不够深入。

因此 v2 的升级方向明确为：

不再只做 2D 重排，而是直接改写扫描序列本身。

### 3.2 核心改动

v2 的核心变化是：

1. 从 2D 区域重排升级为 1D 区域优先扫描。
2. 新增自定义 ig_cross_scan / ig_cross_merge。
3. 真正控制 scan sequence 的构造方式。
4. Stage 4 only 注入，降低实验变量。
5. 默认采用 descending-only。
6. guidance_scale 设为 0.1。
7. 前景头升级为 large-kernel predictor。

### 3.3 v2 的理论优势

相较 v1，v2 更接近最初设想：

1. 重要性不再只影响空间摆放，而是真正影响扫描顺序。
2. scan + merge 的逆过程可验证。
3. 可以清楚解释“先扫谁、后扫谁”。

### 3.4 v2 正式训练后的主要问题

虽然 v2 比 v1 明显更合理，也确实带来了性能提升，但训练与可视化暴露了新的瓶颈：

1. importance 图并没有稳定学成“前景优先图”。
2. 一些高优先区域仍然是停车场、空地、路面等背景区域。
3. region score 往往偏高，很多区域集中在较高分段。
4. 对小车、小船等小目标的敏感性不足。

### 3.5 v2 的问题归因

对 v2 的复盘后，主要问题被归纳为以下几类：

1. 没有显式前景监督。
2. importance 头只能依赖检测损失的间接梯度。
3. 区域评分采用纯均值池化，小目标容易被背景淹没。
4. 排序步骤是离散 argsort，纠偏能力有限。
5. guidance modulation 虽然存在，但不是主因。

### 3.6 结论

v2 证明了：

1. 真正的 1D priority scan 是必要的。
2. 仅靠隐式监督不足以让 importance 学成稳定前景语义。

因此，v2 是方法层面成立、监督层面不足的关键过渡版本。

---

## 4. v3：增加显式前景辅助监督版本

### 4.1 为什么从 v2 升级到 v3

v2 的最大问题不是扫描顺序设计错误，而是 importance head 缺少直接学习目标。

具体表现为：

1. importance map 会学到某种 task-useful 偏置。
2. 但这个偏置不等于前景语义。
3. 最终可视化显示其排序不稳定地偏向高响应背景块。

因此 v3 的核心目标非常明确：

给 importance 头增加显式前景监督，让它真正朝“前景感知”方向学习。

### 4.2 核心改动

v3 相比 v2 的新增改动包括：

1. 从 gt instance masks 合成二值前景图 fg_target。
2. importance head 输出与 fg_target 做 BCE 辅助监督。
3. 新增辅助损失项 loss_fg。
4. 保持 Stage 4 only、5x5 区域划分、1D priority scan 框架不变。
5. 不先改区域粒度，先单独验证“显式监督”这一变量是否有效。

### 4.3 v3 的设计原则

v3 这次迭代刻意遵循最小改动原则：

1. 不推翻 v2 的 1D scan 主体。
2. 不同时引入新区域机制。
3. 不先混入多尺度区域评分。
4. 只新增一个监督变量，确保实验结论可解释。

### 4.4 当前实现状态

v3 已完成代码接入：

1. backbone 训练态可接收由 gt_masks 合成的前景监督图。
2. IG 模块内部可计算 loss_fg。
3. detector 训练时会将 loss_fg 并入总损失。
4. 单测、配置构建和 smoke test 已通过。

### 4.5 v3 正式训练结果

v3 已完成正式 12 epoch 训练验证，最终结果如下：

1. bbox mAP = 0.392
2. segm mAP = 0.370
3. segm mAP_s = 0.236

与前期观察一致，v3 带来了两个清晰结论：

1. importance 可视化显著改善，前景区域尤其是大车密集区域被更稳定地高亮。
2. 但最终 segm mAP 相比 VMamba 2241 baseline 几乎没有提升，说明显式 supervision 修复了“importance 学不准”，却没有进一步解决“importance 如何真正转化为分割收益”。

### 4.6 v3 的复盘结论

v3 训练后的关键判断已经发生变化：

1. 当前瓶颈不再主要是 supervision 缺失。
2. 重要性图已经学得更像前景语义图。
3. 真正的问题更可能是“importance 的注入方式”而不是“importance 的预测质量”。
4. v2/v3 的硬 1D priority reorder scan 仍可能破坏 SSM 所依赖的空间连续状态演化。

因此，v3 的最终结论不是“前景监督无效”，而是：

显式前景监督是必要的，但它不足以单独解决硬重排带来的结构性瓶颈。

## 5. 版本演进总表

| 版本 | 核心改动 | 主要优点 | 主要问题 | 结论 |
|---|---|---|---|---|
| v1 | 区域 2D 重排 | 初步引入区域重要性 | selective scan 会破坏重排效果 | 路径偏弱，不能代表真正 priority scan |
| v2 | 区域优先级 1D 重排 | 真正控制 scan sequence | 无显式监督，小目标被区域均值淹没 | 方法成立，但监督不足 |
| v3 | 在 v2 上增加显式前景辅助 loss | 给 importance 明确前景学习目标 | importance 学准了，但指标几乎未涨 | 监督问题被基本排除 |
| v4 | 恢复官方 scan 路径，importance 调制 z 门控 | 不打断 SSM 连续建模，引入前景感知选择性门控 | 待验证 | 当前下一主线 |

## 6. 关于区域粒度的当前判断

当前阶段不建议在 v3 中同时改区域粒度，原因如下：

1. 现在最需要先验证的是显式监督是否能纠正 importance 学习方向。
2. 如果同时改区域粒度，会混淆 v3 的实验结论。
3. 当前 5x5 区域虽然仍然偏粗，但尚可作为 supervision-only 实验的稳定基线。

因此当前策略是：

v3 先固定 5x5，不改区域划分；若效果仍不理想，再进入 v4。

---

## 7. v4：恢复官方 scan 路径 + importance 调制 z 门控

### 7.1 为什么从 v3 升级到 v4

v3 已经证明：

1. importance head 在显式前景监督下能够学到更可信的前景响应。
2. 但硬重排 scan 仍未把这种前景响应稳定转化为更高的 segm mAP。
3. 继续围绕 reorder 机制扩大到更多 stage 或更大 backbone，风险大于收益。

因此 v4 的升级方向从“继续改排序”切换为：

保留官方 VMamba 的 scan sequence，不再打乱 token 拓扑，而是把 importance 当作连续调制信号注入 z 门控。

### 7.2 v4 的核心思路

v4 的设计原则是：

1. 恢复官方 scan 路径，避免硬重排破坏 SSM 连续性。
2. 恢复带 z 的 SS2D 变体，而不是继续使用 v05_noz。
3. importance 不再参与排序，而是调制 z 分支的空间门控强度。
4. 保留 v3 中已经验证有效的前景 BCE 辅助监督。

其目标可表述为：

让 importance 从“重排控制信号”转变为“前景感知选择性门控信号”。

### 7.3 v4 的当前实现路线

当前已确定并开始实现的 v4-a 版本为：

1. 使用官方 `v05` scan 路径。
2. 在 Stage 3 和 Stage 4 中替换为 IGSS2D 的 `z_gate` 模式。
3. `fg_head` 在卷积激活后的主分支特征上预测 importance。
4. 保留 `loss_fg` 监督。
5. 在最终 `y * z` 之前，将 z 调制为 `z * (1 + gamma * importance)`。

对应的最小实验目标是回答：

在不改变官方 scan 路径的前提下，foreground-aware z gate 是否比硬 priority reorder 更稳、更有效。

当前优先选择 Stage 3+4 联合而不是 Stage 4 only，原因是：

1. z 门控调制属于连续增益控制，不会像硬重排那样直接破坏扫描拓扑。
2. Stage 3 具有更高的空间分辨率，更可能影响中小目标与实例边界。
3. Stage 4 提供更强的高层语义偏置，Stage 3+4 联合更符合实例分割的多尺度需求。
4. 因此 v4 的第一版就应优先验证“连续调制 + 多阶段介入”的组合，而不是沿用 v3 的 Stage 4 only 保守设定。

### 7.4 是否恢复原来的 VMamba 扫描路径

是。

这是 v4 与 v2/v3 的关键分界：

1. v2/v3 直接介入 scan sequence。
2. v4 不再改写 scan path，而是恢复官方 VMamba 的 cross scan + cross merge。
3. 研究重点转为“importance 注入位置”而不是“importance 排序路径”。

这一步不是回退，而是对 v3 结果的针对性修正。

### 7.5 v4 后续分支规划

在 v4-a 的 z 门控调制完成后，再考虑后续两个分支：

1. v4-b：稳定化 z 门控调制
2. v5：importance 调制 dt

当前不建议把三种方案一次性叠加，原因是：

1. 会破坏消融可解释性。
2. 难以判断提升到底来自哪一层调制。
3. dt 调制需要单独验证对 SSM 离散化步长的影响，应后置。

### 7.6 v4-a 训练复盘与问题定位

v4-a 完整训练后，出现了与最初预期不一致的现象：

1. importance 可视化明显优于 v2/v3，前景区域更稳定、更干净。
2. 但最终 epoch 12 指标仅为 bbox mAP 0.280 / segm mAP 0.267。
3. 与此同时，训练中期并非一直低迷，曾在更早阶段达到更高点。

当前确认到的关键日志事实是：

1. v4-a 在正式 run 中的较优阶段出现在 epoch 6，达到 bbox mAP 0.319 / segm mAP 0.315。
2. 随后从 epoch 7 开始逐步退化，到 epoch 12 回落至 0.267。
3. 因此 v4-a 更像“后期训练稳定性不足”而不是“理论方向完全失效”。

基于代码与日志复盘，v4-a 暴露出两个高优先问题：

1. `fg_head` 使用 4 层 `BatchNorm2d`，但当前 batch size 仅为 2，极易导致训练态与验证态统计不稳定。
2. z 门控采用 `z * (1 + gamma * importance)` 的单边增强形式，前景可以被放大，但背景不会被显式抑制。

因此，v4-a 的结论不是“foreground-aware z gate 无效”，而是：

当前这版 z 门控实现仍存在训练稳定性与门控表达能力上的具体缺陷，需要先做一次最小代价修正版验证。

### 7.7 v4-b：GN 前景头 + 安全双向 z 门控

v4-b 的目标非常明确：

1. 不推翻 v4-a 恢复官方 scan 路径的总体方向。
2. 只修复已确认的两个实现级高风险点。
3. 用最小改动判断 z 门控路线是否仍值得继续保留。

#### 7.7.1 核心改动

v4-b 相比 v4-a 新增两项修改：

1. 将 `fg_head` 中的 4 层 `BatchNorm2d` 替换为 `GroupNorm`。
2. 将 z 门控由单向增强改为安全双向调制。

对应的门控形式从：

`z * (1 + gamma * importance)`

改为：

`z * (1 + 0.5 * tanh(gamma) * (2 * importance - 1))`

其中安全设计包括：

1. `importance` 先中心化到 `[-1, 1]`，从而让前景增强、背景抑制成为可能。
2. `gamma` 不再只做非负 clamp，而是通过 `tanh` 转换为有界双向门控强度。
3. 额外乘以 `0.5` 控制最大调制幅度，使门控系数稳定在 `[0.5, 1.5]`，避免训练初期大面积 token 被过度压制。

#### 7.7.2 设计动机

这两个改动分别对应两个具体问题：

1. GN 用于解决小 batch 下 BN 统计不稳定，降低训练态与验证态分布漂移。
2. 双向 z 门控用于让 importance 真正承担“前景-背景差异化调制”而非单边增益。

#### 7.7.3 当前定位

v4-b 是一个最小修正版，而不是新的大版本主线。

它的实验目的不是直接追求最终最优结果，而是先回答：

1. 在修复 BN 与单边门控后，z-gate 路线是否仍然具备继续优化的价值。
2. 如果 v4-b 仍明显不理想，再进入 v5 的 dt 调制分支。

#### 7.7.4 v4-b 完整训练后的结论

v4-b 的完整 12 epoch 训练已经给出比较明确的结论：

1. 它成功修复了 v4-a 的数值不稳定问题。
2. 它证明“恢复官方 scan 路径 + importance 连续调制”这条大方向本身是成立的。
3. 但它也基本说明，继续把主要研发精力投入在 z 分支细修上，性价比已经不高。

从结果上看：

1. v4-b 的最终结果为 bbox mAP 0.390 / segm mAP 0.363。
2. 相比 v4-a 的 0.267，有明显恢复，说明稳定化修复是必要且有效的。
3. 但它仍未超过 RS-LightMamba 2241 baseline 的 segm mAP 0.371，也没有超过 v2/v3 的 0.370 水平。
4. 后期曲线在 epoch 9 到 12 之间基本进入平台区，说明这不是“再多训一点就会自然追上”的问题。

结合日志、可视化与 gate scale 检查，可以把 v4-b 的核心认识总结为：

1. importance 已经能学到较稳定的前景区域响应。
2. 但这些响应更接近“前景显著性”而不是“对实例分割 mAP 最有用的调制信号”。
3. z 门控虽然能做连续调制，但它位于输出门控位置，作用更像后端筛选，未必是影响 SSM 状态更新的最佳注入点。
4. 因此 v4-b 的价值主要在于完成了一次路线去噪：证明问题不再是“importance 完全学不会”，而更可能是“importance 注入位置不够对”。

这意味着从 v4-b 往后，研究重点应正式从“继续打磨 importance 预测精度”切换为：

先验证 SSM 内部的 dt 调制机制本身是否比 z 调制更有效。

---

## 8. v5a：降低 importance 目标，转向粗粒度区域指示 + dt 调制

### 8.1 为什么从 v4-b 升级到 v5a

到 v4-b 为止，已经可以比较确定一件事：

当前 Stage 3/4 的分辨率，不适合再把 importance 当作“逐像素精确前景预测器”来持续打磨。

原因有三：

1. 这不是 FG-IG-Scan 的第一研究目标。论文真正要解决的是 SSM 在遥感前景稀疏场景下的状态更新问题，而不是把 importance head 本身做成一个小分割器。
2. Stage 3/4 的空间分辨率存在硬上限，小目标和边缘稀疏实例很难被精确描成像素级 mask。
3. v4-b 已经说明，哪怕 importance 可视化更“像前景”，也不自动等价于最终 segm mAP 提升。

因此 v5a 的思路是明确降级 importance 的职责：

不再把它当作像素级前景真值拟合器，而是把它当作“粗粒度前景区域指示信号”。

也就是说，v5a 关心的不是：

每个小车的每个像素是否都被高亮。

而是：

1. 目标密集区域应该整体获得更高响应。
2. 混合区域应该获得中等响应。
3. 纯背景区域应该保持低响应。

只要这一点成立，importance 就已经足够为 dt 调制提供有效先验。

### 8.2 v5a 的核心原理

v5a 的关键变化不是“把 importance 做得更准”，而是“把 importance 的作用位置换到更贴近 SSM 状态更新的地方”。

我们将 v4-b 的 z 调制切换为 dt 调制，核心理解如下：

1. z 调制发生在输出门控位置，更像对已经形成的响应做乘性筛选。
2. dt 则直接参与 selective scan 的离散化步长，决定当前 token 的新信息写入强度。
3. 因此在遥感前景稀疏场景下，让前景区域对应的 dt 略大，可以使 SSM 在这些区域更积极地写入新信息。
4. 对纯背景位置，若 importance 近似为 0，则 dt 调制退化为恒等映射，不会比 baseline 更差。

这正对应 v5a 的设计目标：

用“够好”的粗粒度 foreground cue，驱动“更靠近 SSM 内部”的差异化状态更新。

### 8.3 为什么 v5a 先不改监督目标

v5a 明确采用“先验证机制，再优化信号”的顺序。

因此在 v5a 中：

1. 保留 v4-b 已验证稳定的 GroupNorm foreground head。
2. 保留当前 union mask + BCE 的辅助监督。
3. 保留 Stage 3+4 的插入位置。
4. 只改变 importance 的注入位置，从 z 改为 dt。

这样做的原因是：

1. 需要先回答“dt 调制本身有没有用”。
2. 如果连当前这份并不完美但方向正确的 importance 信号都不能通过 dt 调制带来增益，那么优先矛盾就不在 target 设计。
3. 只有在 v5a 验证出正向增益后，再去做小目标友好的 supervision refinement 才有意义。

换句话说，v5a 不是 final refinement，而是一次关键的机制判别实验。

### 8.4 v5a 的最小实现原则

为了保证结论可解释，v5a 严格遵守最小改动原则：

1. `forward_type` 回到 `v05_noz`，不再恢复 z 分支。
2. `ig_mode` 切换为新的 `dt_gate`。
3. 继续在 Stage 3 和 Stage 4 上接入 IG 模块。
4. `fg_head` 继续使用 v4-b 的 `GroupNorm` 版本。
5. `loss_fg` 继续使用当前 union mask + BCE。
6. 新增一个可学习标量 `dt_scale`，初始化为 0.03。
7. 在官方 cross scan 路径下，对有效 dt 做逐位置乘性调制。

对应的最小形式为：

`dt_eff = softplus(dts + bias)`

`dt_eff' = dt_eff * (1 + beta * importance)`

这里强调的是“对有效 dt 乘法调制”，而不是“对 raw dts 乘法调制”。

这样处理主要出于两点考虑：

1. selective scan 内部真正起作用的是 `softplus(dts + bias)` 之后的有效 dt，而不是 raw `dts` 本身。
2. 若直接对 raw `dts` 做乘法放大，当某些通道的 raw `dts` 为负时，乘以大于 1 的系数反而可能让有效 dt 变小；而在有效 dt 空间做乘法，则能同时保证前景区域单调增强和各通道相对变化一致。

### 8.5 v5a 的实现流程

v5a 的前向流程可概括为：

1. 输入特征先经过与 v4-b 相同的前置投影、卷积和激活。
2. foreground head 在该特征上预测一个粗粒度 importance map。
3. 主分支继续走官方 `cross_scan`，不再做区域重排。
4. 将 importance 通过同样的 `cross_scan` 展平到四个扫描方向。
5. 在送入 `selective_scan_fn` 之前，先显式计算有效 dt，再施加 `(1 + beta * importance)` 的乘性缩放。
6. 之后以 `delta_softplus=False` 的形式将该有效 dt 送入 selective scan，其余 cross merge 和输出归一化全部保持官方路径。

这版设计回答的核心问题是：

如果 importance 只作为一个粗粒度前景区域提示，而不是精细 mask，那么把它作用到 dt 上，是否就足以为遥感实例分割带来稳定收益。

### 8.6 v5a 的当前实验定位

v5a 是 v5 系列的第一版，不承担一次性解决所有小目标问题的任务。

它的职责只有一个：

验证“粗粒度 foreground cue + dt 调制”是否比“粗粒度 foreground cue + z 调制”更符合 RS-LightMamba 的主创新方向。

因此 v5a 跑完后，判断逻辑应保持简单：

1. 若 v5a 超过 baseline 0.371，则说明主路线正式切到 dt 分支。
2. 若 v5a 接近 baseline，则优先微调 `dt_scale` 或 stage 组合。
3. 若 v5a 明显低于 v4-b，则再回头审视当前 importance 信号是否过弱，或 LightMamba 容量是否成为主瓶颈。

在 v5a 之前，不再继续把大量时间投入到 importance target 的细化上；

在 v5a 之后，若 dt 路线成立，再按优先级考虑：

1. 多尺度 importance
2. 更平滑的小目标下采样策略
3. instance-balanced target

### 8.7 v5a 完整训练总结

v5a 已经完成了它的阶段性使命：

1. 它证明了把 importance 的主注入位置从 z 切到 dt 是有效的。
2. 它说明当前研究主线应坚定切换到 dt 调制，而不是继续回到 z-gate 的细修。
3. 它也暴露出一个新的实现级瓶颈：dt 调制已经方向正确，但强度利用还不充分。

v5a 的最终结果为：

1. bbox mAP 0.394
2. segm mAP 0.369
3. segm APs 0.231
4. segm APm 0.442
5. segm APl 0.529

对比已有版本可以得到三个清晰结论：

1. 相比 v4-b 的 segm 0.363，v5a 提升到 0.369，说明 dt 主线优于 z-gate 主线。
2. 相比 RS-LightMamba 2241 baseline 的 segm 0.371，v5a 只差 0.002，说明它已经基本追平 baseline。
3. APs 从 v4-b 的 0.223 提升到 0.231，甚至略高于 baseline 的 0.229，这说明当前的第一矛盾已经不再是“小目标完全学不到”。

因此，v5a 的历史定位应写得很明确：

它不是终版，但它已经完成了“证明 dt 路线成立”的任务。

### 8.8 为什么 v5a 还差最后 0.002

v5a 训练完后，对 epoch 12 checkpoint 中 5 个 IG block 的 `dt_scale` 检查表明：

1. 其中两个 block 的调制强度明显学大，约为 0.1566 和 0.0908。
2. 另外三个 block 的 `dt_scale` 落入负值附近。
3. 由于 v5a 当时采用 `clamp(min=0)`，这三个 block 在前向中等效为调制强度 0。

这意味着：

1. v5a 虽然在结构上给 5 个 IG block 都接入了 dt 调制。
2. 但在最终训练状态里，真正处于活跃调制状态的只有 2 个 block。
3. 其余 3 个 block 等效“熄火”，从而限制了 v5a 的上限释放。

这也解释了为什么 v5a 已经明显优于 v4-b，却仍只差 baseline 最后一点点。

问题不再是“dt 这条路线对不对”，而是：

当前 dt 调制强度参数化方式，会把一部分 block 训练成永久失活状态。

## 9. v5b：修复 dt_scale 死区 + 提高初始化强度 + area 下采样

### 9.1 为什么从 v5a 升级到 v5b

v5b 的目标不是重新换方向，而是在已经成立的 dt 主线上做最小但关键的加强。

升级原因有三：

1. v5a 已证明 dt 路线优于 v4-b，因此不需要再争论“是否继续调 dt”。
2. checkpoint 检查显示 v5a 有 3 个 block 因 `clamp(min=0)` 落入死区，导致有效 dt 调制覆盖率不足。
3. 当前 fg supervision 仍使用 `nearest` 下采样，这对小目标 union mask 过于粗糙，可以用极低风险的方式进一步改进。

因此 v5b 的任务可以概括为：

让已经被验证有效的 dt 主线，从“部分 block 生效”升级为“所有 block 都可持续参与调制”。

### 9.2 v5b 的三项改动

v5b 只做三项修改，其余设计保持 v5a 不变。

#### 9.2.1 改动一：dt_scale 从 clamp 变为 softplus 正值重参数化

v5a 的问题在于：

`safe_dt_scale = clamp(dt_scale, min=0)`

这种写法一旦把某个 block 的 `dt_scale` 训练到负区间，就会同时导致：

1. 前向里该 block 的调制强度被硬截为 0。
2. 负区间梯度消失，参数难以自行回到有效区域。

v5b 采用更稳妥的写法：

1. 存储一个 raw 标量参数 `dt_scale_raw`。
2. 前向时用 `softplus(dt_scale_raw)` 得到真正参与调制的正值强度。

即：

`beta = softplus(beta_raw)`

这样可以保证：

1. dt 调制强度始终为正。
2. 即使 raw 参数被推到较小值，梯度仍然存在，不会形成硬死区。
3. 所有 IG block 都保留继续学习调制强度的机会。

#### 9.2.2 改动二：dt_scale 初始化从 0.03 提高到 0.06

v5a 的初值 0.03 已证明是稳定的，但它偏保守。

结合 v5a 末期已有 block 自发学到 0.09 甚至 0.15 的事实，可以合理推断：

1. 当前机制允许更大的 dt 调制强度。
2. 初值过小会让一部分 block 在训练早期过于接近“弱调制”状态。

因此 v5b 将有效初值直接提高到 0.06。

这一步的目的不是激进放大，而是：

让所有 IG block 在训练起点就进入“可感知的 dt 差异化调制区间”。

#### 9.2.3 改动三：fg_target 下采样由 nearest 改为 area，并重新二值化

v5a 中，foreground supervision 的下采样仍采用 `nearest`。

这会带来一个简单但实际的问题：

1. 当原图 union mask 下采样到 Stage 3/4 分辨率时，极小目标可能被直接采样丢失。
2. importance head 学到的 foreground cue 会因此对小目标不够友好。

v5b 的修正方法为：

1. 先用 `area` 做下采样，保留每个低分辨率 cell 中的前景占比信息。
2. 再做一次 `> 0` 的重二值化，把“该区域是否出现过前景”重新转成二值目标。

这样做的好处是：

1. 不改变 union mask + BCE 这条总体监督路线。
2. 但能显著减少小目标在下采样阶段被完全抹掉的概率。
3. 改动很小，风险很低，适合作为 v5b 的配套增强。

### 9.3 v5b 为什么暂时不做的事情

为了保持 v5b 的可解释性，以下内容明确后置：

1. 不恢复 z 分支。
2. 不做 dt + 输出双调制。
3. 不做 instance-balanced target。
4. 不做 soft target。

理由很简单：

当前最优先的问题已经足够明确，是 dt 调制强度参数化与小目标 supervision 下采样方式，而不是再次扩大变量集合。

### 9.4 v5b 的实现清单

v5b 相比 v5a 的实际修改只应包括：

1. `ig_ss2d.py` 中将 `dt_scale` 改为 `dt_scale_raw -> softplus(dt_scale_raw)`。
2. v5b 配置中将 `ig_dt_scale` 有效初值设为 0.06。
3. `fg_ig_scan.py` 中将 fg_target 插值改为 `area`，并在下采样后重二值化。

其余全部保持和 v5a 一致。

### 9.5 v5b 的当前实验目标

v5b 不是新的机制版，而是 v5 主线上的强度校准版。

它要回答的问题非常聚焦：

在不引入额外调制分支、不改 supervision 主框架的前提下，仅通过修复 dt_scale 死区并提高起始强度，是否就足以让 v5 系列超过 0.375。
