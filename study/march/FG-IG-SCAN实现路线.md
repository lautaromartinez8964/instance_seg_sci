# FG-IG-Scan 实现路线与版本演进记录

## 0. 文档目的

这份文档用于记录 FG-IG-Scan 创新点从提出、实现、暴露问题到迭代修正的全过程，作为后续论文写作、组会汇报、实验回溯和版本管理依据。

从本版本开始，FG-IG-Scan 每次发生方法级演进，都单独记为一个版本：v1、v2、v3、v4...

要求：

1. 每个版本必须写清楚核心改动。
2. 每个版本必须写清楚为什么要改。
3. 每个版本必须记录暴露出的关键问题。
4. 每个版本必须说明是否进入正式训练验证。

---

## 1. 创新点目标

FG-IG-Scan 的目标不是简单“改一下扫描顺序”，而是解决遥感实例分割中以下核心矛盾：

1. 前景区域稀疏，固定扫描顺序会让 SSM 隐状态首先被大量背景占据。
2. 小目标在长序列建模中容易被大面积背景稀释。
3. 遥感图像中的目标分布具有明显空间非均匀性，固定扫描路径无法体现“先处理关键区域”的需求。

因此，FG-IG-Scan 的主线目标始终是：

让扫描顺序从固定路径，演进为受前景重要性引导的动态优先级扫描。

---

## 2. v1：区域 2D 重排版本

### 2.1 核心思路

v1 的出发点是：

1. 先预测区域重要性。
2. 在 2D 空间上对区域进行重排。
3. 再把重排后的特征送入官方 VMamba 的 selective scan 路径。

简化理解：

先改空间布局，再交给原始扫描器去扫。

### 2.2 实现特征

v1 主要采用的是：

1. 区域级重要性预测。
2. 2D 区域 permute / inverse permute。
3. 外部包裹官方 forward_core，不直接改内部 cross_scan。

### 2.3 暴露的问题

这是 FG-IG-Scan 第一次真正暴露核心缺陷的版本。

关键问题在于：

1. 它不是真正的 1D 扫描顺序控制。
2. 它只是 2D 区域重排，但后面仍然调用官方固定 cross_scan。
3. 官方 4-path 扫描与 merge 会冲淡区域优先级的效果。
4. 因此，重排后的“优先顺序”并没有被充分保留下来。

### 2.4 结论

v1 证明了一件事：

只在 2D 布局层面重排，而不真正控制 scan sequence，本质上是不够的。

因此 v1 被判定为：

概念上接近目标，但实现路径偏弱，无法充分体现“区域优先扫描”的真实作用。

---

## 3. v2：区域优先级 1D 重排版本

### 3.1 为什么从 v1 升级到 v2

v1 的主要问题不是“没有 importance”，而是 importance 对真正扫描序列的控制不够深入。

因此 v2 的升级方向明确为：

不再只做 2D 重排，而是直接改写扫描序列本身。

### 3.2 核心改动

v2 的核心变化是：

1. 从 2D 区域重排升级为 1D 区域优先扫描。
2. 新增自定义 ig_cross_scan / ig_cross_merge。
3. 真正控制 scan sequence 的构造方式。
4. Stage 4 only 注入，降低实验变量。
5. 默认采用 descending-only。
6. guidance_scale 设为 0.1。
7. 前景头升级为 large-kernel predictor。

### 3.3 v2 的理论优势

相较 v1，v2 更接近最初设想：

1. 重要性不再只影响空间摆放，而是真正影响扫描顺序。
2. scan + merge 的逆过程可验证。
3. 可以清楚解释“先扫谁、后扫谁”。

### 3.4 v2 正式训练后的主要问题

虽然 v2 比 v1 明显更合理，也确实带来了性能提升，但训练与可视化暴露了新的瓶颈：

1. importance 图并没有稳定学成“前景优先图”。
2. 一些高优先区域仍然是停车场、空地、路面等背景区域。
3. region score 往往偏高，很多区域集中在较高分段。
4. 对小车、小船等小目标的敏感性不足。

### 3.5 v2 的问题归因

对 v2 的复盘后，主要问题被归纳为以下几类：

1. 没有显式前景监督。
2. importance 头只能依赖检测损失的间接梯度。
3. 区域评分采用纯均值池化，小目标容易被背景淹没。
4. 排序步骤是离散 argsort，纠偏能力有限。
5. guidance modulation 虽然存在，但不是主因。

### 3.6 结论

v2 证明了：

1. 真正的 1D priority scan 是必要的。
2. 仅靠隐式监督不足以让 importance 学成稳定前景语义。

因此，v2 是方法层面成立、监督层面不足的关键过渡版本。

---

## 4. v3：增加显式前景辅助监督版本

### 4.1 为什么从 v2 升级到 v3

v2 的最大问题不是扫描顺序设计错误，而是 importance head 缺少直接学习目标。

具体表现为：

1. importance map 会学到某种 task-useful 偏置。
2. 但这个偏置不等于前景语义。
3. 最终可视化显示其排序不稳定地偏向高响应背景块。

因此 v3 的核心目标非常明确：

给 importance 头增加显式前景监督，让它真正朝“前景感知”方向学习。

### 4.2 核心改动

v3 相比 v2 的新增改动包括：

1. 从 gt instance masks 合成二值前景图 fg_target。
2. importance head 输出与 fg_target 做 BCE 辅助监督。
3. 新增辅助损失项 loss_fg。
4. 保持 Stage 4 only、5x5 区域划分、1D priority scan 框架不变。
5. 不先改区域粒度，先单独验证“显式监督”这一变量是否有效。

### 4.3 v3 的设计原则

v3 这次迭代刻意遵循最小改动原则：

1. 不推翻 v2 的 1D scan 主体。
2. 不同时引入新区域机制。
3. 不先混入多尺度区域评分。
4. 只新增一个监督变量，确保实验结论可解释。

### 4.4 当前实现状态

v3 已完成代码接入：

1. backbone 训练态可接收由 gt_masks 合成的前景监督图。
2. IG 模块内部可计算 loss_fg。
3. detector 训练时会将 loss_fg 并入总损失。
4. 单测、配置构建和 smoke test 已通过。

### 4.5 v3 正式训练结果

v3 已完成正式 12 epoch 训练验证，最终结果如下：

1. bbox mAP = 0.392
2. segm mAP = 0.370
3. segm mAP_s = 0.236

与前期观察一致，v3 带来了两个清晰结论：

1. importance 可视化显著改善，前景区域尤其是大车密集区域被更稳定地高亮。
2. 但最终 segm mAP 相比 VMamba 2241 baseline 几乎没有提升，说明显式 supervision 修复了“importance 学不准”，却没有进一步解决“importance 如何真正转化为分割收益”。

### 4.6 v3 的复盘结论

v3 训练后的关键判断已经发生变化：

1. 当前瓶颈不再主要是 supervision 缺失。
2. 重要性图已经学得更像前景语义图。
3. 真正的问题更可能是“importance 的注入方式”而不是“importance 的预测质量”。
4. v2/v3 的硬 1D priority reorder scan 仍可能破坏 SSM 所依赖的空间连续状态演化。

因此，v3 的最终结论不是“前景监督无效”，而是：

显式前景监督是必要的，但它不足以单独解决硬重排带来的结构性瓶颈。

## 5. 版本演进总表

| 版本 | 核心改动 | 主要优点 | 主要问题 | 结论 |
|---|---|---|---|---|
| v1 | 区域 2D 重排 | 初步引入区域重要性 | selective scan 会破坏重排效果 | 路径偏弱，不能代表真正 priority scan |
| v2 | 区域优先级 1D 重排 | 真正控制 scan sequence | 无显式监督，小目标被区域均值淹没 | 方法成立，但监督不足 |
| v3 | 在 v2 上增加显式前景辅助 loss | 给 importance 明确前景学习目标 | importance 学准了，但指标几乎未涨 | 监督问题被基本排除 |
| v4 | 恢复官方 scan 路径，importance 调制 z 门控 | 不打断 SSM 连续建模，引入前景感知选择性门控 | 待验证 | 当前下一主线 |

## 6. 关于区域粒度的当前判断

当前阶段不建议在 v3 中同时改区域粒度，原因如下：

1. 现在最需要先验证的是显式监督是否能纠正 importance 学习方向。
2. 如果同时改区域粒度，会混淆 v3 的实验结论。
3. 当前 5x5 区域虽然仍然偏粗，但尚可作为 supervision-only 实验的稳定基线。

因此当前策略是：

v3 先固定 5x5，不改区域划分；若效果仍不理想，再进入 v4。

## 7. v4：恢复官方 scan 路径 + importance 调制 z 门控

### 7.1 为什么从 v3 升级到 v4

v3 已经证明：

1. importance head 在显式前景监督下能够学到更可信的前景响应。
2. 但硬重排 scan 仍未把这种前景响应稳定转化为更高的 segm mAP。
3. 继续围绕 reorder 机制扩大到更多 stage 或更大 backbone，风险大于收益。

因此 v4 的升级方向从“继续改排序”切换为：

保留官方 VMamba 的 scan sequence，不再打乱 token 拓扑，而是把 importance 当作连续调制信号注入 z 门控。

### 7.2 v4 的核心思路

v4 的设计原则是：

1. 恢复官方 scan 路径，避免硬重排破坏 SSM 连续性。
2. 恢复带 z 的 SS2D 变体，而不是继续使用 v05_noz。
3. importance 不再参与排序，而是调制 z 分支的空间门控强度。
4. 保留 v3 中已经验证有效的前景 BCE 辅助监督。

其目标可表述为：

让 importance 从“重排控制信号”转变为“前景感知选择性门控信号”。

### 7.3 v4 的当前实现路线

当前已确定并开始实现的 v4-a 版本为：

1. 使用官方 `v05` scan 路径。
2. 在 Stage 3 和 Stage 4 中替换为 IGSS2D 的 `z_gate` 模式。
3. `fg_head` 在卷积激活后的主分支特征上预测 importance。
4. 保留 `loss_fg` 监督。
5. 在最终 `y * z` 之前，将 z 调制为 `z * (1 + gamma * importance)`。

对应的最小实验目标是回答：

在不改变官方 scan 路径的前提下，foreground-aware z gate 是否比硬 priority reorder 更稳、更有效。

当前优先选择 Stage 3+4 联合而不是 Stage 4 only，原因是：

1. z 门控调制属于连续增益控制，不会像硬重排那样直接破坏扫描拓扑。
2. Stage 3 具有更高的空间分辨率，更可能影响中小目标与实例边界。
3. Stage 4 提供更强的高层语义偏置，Stage 3+4 联合更符合实例分割的多尺度需求。
4. 因此 v4 的第一版就应优先验证“连续调制 + 多阶段介入”的组合，而不是沿用 v3 的 Stage 4 only 保守设定。

### 7.4 是否恢复原来的 VMamba 扫描路径

是。

这是 v4 与 v2/v3 的关键分界：

1. v2/v3 直接介入 scan sequence。
2. v4 不再改写 scan path，而是恢复官方 VMamba 的 cross scan + cross merge。
3. 研究重点转为“importance 注入位置”而不是“importance 排序路径”。

这一步不是回退，而是对 v3 结果的针对性修正。

### 7.5 v4 后续分支规划

在 v4-a 的 z 门控调制完成后，再考虑后续两个分支：

1. v4-b：输入 + 输出双调制
2. v5：importance 调制 dt

当前不建议把三种方案一次性叠加，原因是：

1. 会破坏消融可解释性。
2. 难以判断提升到底来自哪一层调制。
3. dt 调制风险明显高于 z 门控调制，应后置。
