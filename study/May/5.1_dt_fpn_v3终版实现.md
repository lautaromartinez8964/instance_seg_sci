# 5.1 DT-FPN v3 终版实现手册

## 1. 版本定位

`DT-FPN v3` 是当前 `RTMDet-Ins-Tiny` 路线上的最终 DT 引导版本。

它的设计目标不是重写整个 RTMDet，而是坚持一个非常克制的原则：

- backbone 不动
- bbox / mask head 主体不动
- 只在 neck 的 top-down 融合口注入 DT 先验
- 用一个轻量辅助损失把这个先验学稳

这套设计背后的直觉很直接：

- 深层特征语义强，但边界容易糊
- 浅层特征边界细，但语义弱、噪声大
- 密集砾石最怕这两种信息在 FPN 里被“平均掉”

所以 v3 不再让 FPN 盲目把上下层特征直接拼起来，而是先让网络自己预测一张 DT 图，再用这张 DT 图生成门控权重，决定每个空间位置到底更该保留浅层细节，还是更该相信深层语义。

一句话概括：

`DT-FPN v3 = 在 RTMDet-Ins 的 FPN 融合口加入 shared DT-guided gate，并通过辅助 DT loss 训练这个 gate。`

## 3. 相比原始 RTMDet-Ins 改了哪些

### 3.1 模型结构改动

baseline:

- `RTMDet`
- `CSPNeXtPAFPN`

v3:

- `RTMDetWithAuxNeck`
- `DTCSPNeXtPAFPN`

也就是说，v3 保留了原始 RTMDet 的 backbone、head、train/test cfg，总体框架没有推翻，只是把 neck 换成了带 DT 引导能力的版本，并给 detector 加了一层“辅助监督转发器”。

### 3.2 数据流改动

baseline 的训练 pipeline 只加载图像、bbox、mask。

v3 额外插入：

`LoadGravelDistanceTransform`

这样每张训练图除了实例掩码以外，还会额外拿到一张离线生成的 DT 监督图。

### 3.3 训练策略改动

这一点必须写清楚，因为这是导师最容易追问的地方。

v3 并不是“纯 neck-only 对照”。

除了 neck 改了，训练 pipeline 也改了：

- baseline: 前期强增强（Mosaic / MixUp），后期切弱增强
- v3: 全程简化增强，不再沿用这套切换策略

所以实验结论的正确表述应该是：

`Shared DT-FPN v3 的收益来自 DT-guided neck 与训练策略调整的联合作用。`

如果后续要做纯结构归因，需要再补严格对照实验。

## 4. DT GT 是怎么传入模型的

这一部分是整个实现闭环里最重要的工程链路。

### 4.1 离线 DT 文件的位置

v3 配置中：

`distance_root='data/gravel_big_mmdet/auxiliary_labels_coreband'`

实际读取路径为：

`data/gravel_big_mmdet/auxiliary_labels_coreband/<split>/distance_transform/<img_stem>.png`

也就是说，训练 / 验证图像各自对应一张灰度 PNG DT 图。

### 4.2 LoadGravelDistanceTransform 做了什么

`LoadGravelDistanceTransform` 在 `gravel_loading.py` 中完成三件事：

1. 根据当前样本的 `img_path` 自动解析对应 split 和文件名
2. 从磁盘读取灰度 DT PNG
3. 把它写入 `results['gt_seg_map']`

之所以写到 `gt_seg_map`，不是因为它是语义分割任务，而是为了借用 MMDetection 现成的几何增强通路。这样后续的：

- RandomResize
- RandomCrop
- RandomFlip
- Pad

都会自动对这张 DT 图做同样的空间变换，从而保证它和原图始终严格对齐。

### 4.3 PackDetInputs 如何把 DT GT 送进 data_sample

在 `PackDetInputs` 里，如果 `results` 中存在 `gt_seg_map`，就会把它包装成：

`data_sample.gt_sem_seg.sem_seg`

所以到模型侧，DT GT 的正式入口不是 `results['gt_seg_map']`，而是：

`batch_data_samples[i].gt_sem_seg.sem_seg`

这一步非常关键，因为后面 neck 取监督时就是从这里拿的。

### 4.4 RTMDetWithAuxNeck 如何把 GT 交给 neck

`RTMDetWithAuxNeck.loss()` 相比原始 `RTMDet.loss()` 多做了一步：

在 backbone / neck forward 之前，先调用：

`self.neck.set_auxiliary_targets(batch_data_samples, input_shape=..., device=...)`

也就是说，detector 本身并不计算 DT loss，它只是把 GT 转交给 neck，让 neck 自己保存当前 batch 的辅助目标，再在本次 forward 后自行计算辅助损失。

这个设计的好处是：

- 改动局部
- 责任清晰
- 推理阶段不需要额外改 head

## 5. 预测 DT 的头在哪里

### 5.1 核心位置

预测 DT 的头在：

`mmdet/models/necks/dt_cspnext_pafpn.py`

类名：

`DTCSPNeXtPAFPN`

### 5.2 当前终版使用的是 shared 模式

v3 配置中写死：

- `dt_mode='shared'`
- `dt_decoder_source='inputs'`
- `dt_head_channels=128`

这表示当前终版不是每层各预测一张 DT，而是先从 neck 输入特征中解码出一张共享 DT 图，再把它 resize 到各个 guided level 上使用。

### 5.3 shared DT 头的组成

当 `dt_mode='shared'` 时，neck 内部会构建：

- `dt_lateral_adapters`
	- 把每层输入特征先统一投影到 `dt_head_channels`

- `dt_fuse_blocks`
	- 从高层到低层逐级上采样并融合，形成 DT 解码路径

- `dt_predictor`
	- 最后一层 `1x1 conv`，输出单通道 DT 图

如果用做比喻：

- backbone / neck 输入特征像多份不同分辨率的食材
- `dt_lateral_adapters + dt_fuse_blocks` 像一条专门做“边界/结构浓汤”的小厨房
- `dt_predictor` 是最后出汤口，只出一张共享 DT 图

### 5.4 shared DT 图的分辨率

因为当前配置：

`dt_decoder_source='inputs'`

所以 shared DT 解码器直接吃 neck 输入的三层特征，并一路上采样到最高分辨率输入层大小。

在当前 RTMDet-Tiny 配置下，640×640 输入对应的最高 neck 输入层通常是 stride 8，所以最终 shared DT 图本质上是一个高分辨率的结构先验图，然后再 resize 到其它 guided level 使用。

## 6. DT loss 是怎么计算的

### 6.1 GT 提取

`set_auxiliary_targets()` 内部会调用 `_extract_dt_target()`：

1. 从 `data_sample.gt_sem_seg.sem_seg` 取出当前样本的 DT GT
2. 转成 `float32`
3. 如果值域还是 `0~255`，就除以 `255.0`
4. 构造 `valid_mask`

这里的 `valid_mask` 非常关键。

原因是训练 pipeline 里 pad 的 seg 值是 `255`，如果直接除以 255，就会把无效区域错误变成 1.0 监督，这会把边界学习彻底带歪。

所以当前实现里会显式做：

- `dt_target < 254.5` 的位置视为有效像素
- pad / ignore 区域在 loss 计算时忽略

### 6.2 尺寸对齐

`set_auxiliary_targets()` 会把 GT pad 或 resize 到当前 batch 的 `input_shape`。

这样即使前面经过了 crop / pad，neck 中计算 loss 时也能和当前前向输入严格对齐。

### 6.3 loss 公式

当前 v3 的 DT loss 是加权 MSE：

`loss_neck_dt = mean((dt_pred - dt_gt)^2 over valid pixels) * dt_loss_weight`

当前配置：

`dt_loss_weight = 0.2`

也就是说，它不是主任务，而是一个温和的辅助牵引项。

### 6.4 当前配置下监督的是谁

因为当前终版使用 `dt_mode='shared'`，所以实际被监督的是那一张共享 DT 图。

如果以后切到 `per_level` 或 `shared_refined`，代码也支持对多张 DT 图分别监督，但这不是 v3 final 当前启用的路径。

## 7. 预测出来的 DT 如何融合到 FPN

这部分是 v3 的灵魂。

### 7.1 原始 RTMDet 的 top-down 融合

原始 `CSPNeXtPAFPN` 的做法可以粗暴理解成：

- 先把高层特征 reduce
- 上采样
- 和低层特征 concat
- 再过 CSP block 融合

这套做法对普通目标检测足够强，但对密集砾石这种边界细碎、实例挨得很近的任务，有一个天然问题：

`高层语义和低层细节被统一配比地混合，缺少空间位置上的选择性。`

### 7.2 v3 的 gate 机制

在 `forward()` 里，当某个 `target_level` 位于 `guided_levels` 中时，v3 会先拿到该层对应的 DT 图：

- shared 模式：把共享 DT 图 resize 到当前层分辨率

然后计算门控：

`gate = sigmoid(scale_l * dt_l + bias_l)`

其中：

- `scale_l` 和 `bias_l` 是每个 fusion level 的可学习参数
- `dt_l` 是 resize 后的 DT 图

### 7.3 融合公式

得到 gate 后，不再直接把低层和上采样高层暴力拼起来，而是先做加权路由：

`guided_low = feat_low * (1 - gate) + upsample_feat * gate`

这个公式的物理直觉很强：

- `gate` 大的地方，更相信上采样高层语义
- `gate` 小的地方，更保留低层高频细节

也就是说，网络不再被迫用同一个融合比例处理整张图，而是允许“边界区域”和“内部区域”走不同的特征路由。

这正是密集砾石最需要的能力。

### 7.4 当前 v3 是如何指导各层的

当前配置：

`guided_levels=[0,1]`

这意味着当前 v3 只引导 top-down 路径中的两个融合点。

结合 RTMDet-Tiny 的 3 层输入结构，可以理解为：

- 高分辨率层引导开启
- 中分辨率层引导开启
- 最粗层不做 DT 引导

这是非常合理的，因为：

- DT 的价值主要在边界和结构
- 最粗层更多承担全局语义，不值得额外堆 DT 引导

### 7.5 skip_csp_on_guided 的作用

当前配置：

`skip_csp_on_guided=True`

这意味着，在 guided level 上，v3 不再把 `guided_low` 丢回原来的 top-down CSP block 再混一遍，而是直接把门控融合结果当作该层输出。

这个点很关键。

因为如果 gate 刚把“该保的边界细节”和“该信的高层语义”分好了，后面再用一个通用 CSP block 全量搅一遍，门控的收益就容易被稀释。

所以当前终版的策略是：

`先用 gate 做结构化分流，再尽量避免后续模块把这个分流重新洗平。`

## 8. “共享”到底共享了什么

这里非常容易被问到，必须讲清楚。

当前 `shared` 模式里，共享的不是 gate 参数，而是：

`同一张 DT 预测图被多个 guided level 共同使用。`

具体来说：

1. 先从多层输入特征解码出一张 shared DT map
2. 在每个 guided level 上，把这张图 resize 到对应分辨率
3. 再结合该层自己的 `scale_l` 和 `bias_l` 生成本层 gate

所以共享层面是：

- 共享结构先验来源
- 共享 DT 解码器

不共享的层面是：

- 每层 gate 的最终缩放和偏置参数
- 每层最终融合后的特征状态

这就避免了“每层完全独立预测 DT 图”带来的额外参数和额外不稳定，同时也避免“所有层完全同一个 gate”带来的过度僵硬。

这是一个很好的折中。

## 9. forward / backward 的完整训练闭环

为了给导师看得更清楚，可以把一次训练写成下面这条链：

1. 图像、bbox、mask 正常读取
2. `LoadGravelDistanceTransform` 读取离线 DT PNG，写入 `results['gt_seg_map']`
3. 几何增强对图像和 DT GT 同步执行
4. `PackDetInputs` 把 `gt_seg_map` 封装到 `data_sample.gt_sem_seg`
5. `RTMDetWithAuxNeck.loss()` 在 forward 前调用 `neck.set_auxiliary_targets()`
6. backbone 提取多尺度特征
7. `DTCSPNeXtPAFPN.forward()` 解码 shared DT map
8. shared DT map resize 到 guided level，生成 gate
9. gate 引导 top-down 融合，输出新的 FPN 特征
10. bbox / mask head 正常计算主任务 loss
11. neck 根据 shared DT map 与 DT GT 计算 `loss_neck_dt`
12. detector 将 `loss_neck_dt` 拼回总 loss，一起反传

所以 v3 不是额外开了一个大分支去做显式分割任务，而是把 DT 监督嵌进了 neck 的表示学习过程。

## 10. 推理阶段怎么工作

推理阶段没有 DT GT 输入。

但这并不影响 v3 工作，因为：

- DT GT 只在训练时用于监督 shared DT predictor
- 推理时，shared DT map 完全由图像特征自己预测
- 然后照样生成 gate 去引导 FPN 融合

也就是说，v3 的推理路径依然是单模型、单输入图像，不需要额外提供任何辅助标签。

这点很重要，因为它说明：

`DT supervision 是训练期先验，不是推理期负担。`

## 11. 当前终版的实验结果与代价

相对 baseline：

- best segm mAP: `0.272 -> 0.281`，提升 `+0.009`
- Params: `5.615M -> 7.275M`，增加 `+29.6%`
- FLOPs: `11.873G -> 15.989G`，增加 `+34.7%`
- FPS: `156.7 -> 155.1 img/s`，几乎持平
- CUDA Memory: `64MB -> 70MB`

这个 trade-off 很有意思：

- 算量和参数量确实上去了
- 但推理速度几乎没掉
- 精度提升主要体现在中大尺度实例的结构完整性和分离能力上

这说明 v3 的增益来自“结构化特征路由更合理”，而不是靠堆很重的推理分支换精度。

## 12. 这个版本为什么有效

从机制上说，v3 有效的原因主要有四条：

### 12.1 它把 DT 先验放在了最该放的位置

不是塞进最终 head，不是单独再做一个复杂分支，而是直接放在 FPN top-down 融合口。

这里正是“浅层边界”和“深层语义”发生冲突的地方。

### 12.2 它不是硬规则，而是可学习 gate

DT 图本身不直接替代特征，而是通过可学习的 `scale/bias + sigmoid` 变成门控权重。

这样模型保留了足够的自适应空间。

### 12.3 shared 设计控制了复杂度

如果每层都独立预测一张 DT 图，参数和不稳定性都会继续涨。

当前 shared 模式用一张共享 DT 图服务多个 guided level，在开销和效果之间取得了平衡。

### 12.4 辅助损失足够轻

`dt_loss_weight=0.2` 说明 DT supervision 不是压过主任务的强监督，而是一个“把 neck 往结构感知方向推一把”的温和牵引项。

这比很多重型辅助头更稳。

## 13. 当前版本的边界与后续建议

### 13.1 当前结论的边界

必须实事求是：

当前 v3 的收益不能严格归因为 neck 单因素，因为训练增强策略也变了。

所以手册里建议统一使用下面这句表述：

`DT-FPN v3 在 RTMDet-Ins-Tiny 上通过 shared DT-guided top-down fusion 与 DT 辅助监督提升了结构建模能力，并在当前简化增强训练策略下取得了稳定精度增益。`

### 13.2 后续最值得补的三个消融

1. 保持 v3 的简化 pipeline，只把 neck 换回原始 PAFPN
2. 保留 DT loss，去掉 gate 融合
3. 保留 gate 融合，去掉 DT loss

只有这三组补上，才能把“监督贡献”和“路由贡献”真正拆开。

---

## 14. 一句话总结给导师

`DT-FPN v3` 的本质，不是再造一个复杂新模型，而是在 RTMDet-Ins 原有 neck 中加入一条轻量 shared DT 解码路径，用 DT 预测图生成空间门控，引导 top-down 融合时的浅层细节保留与深层语义注入；训练时通过辅助 MSE loss 学稳这张 DT 图，推理时则不增加额外标签输入，从而以可控的参数与 FLOPs 代价，换来更强的密集砾石实例结构建模能力。
